# 모청 서비스 정책 및 개발 기준

본 문서는 모청(청첩장 빌더) 서비스의 일관된 품질을 위해, UI/UX/서비스/백엔드 정책을 정의한다.
실제 코드 변경 시 본 문서를 우선 기준으로 삼고, 필요한 경우 PR에서 변경 이유를 명시한다.

> 서비스 전반의 기준(회원/결제/보안/환경 등)은 `docs/PRODUCT_STANDARD.md`를 최상위로 한다.
> 본 문서는 그 중 UI/UX/옵션 정책의 세부 가이드라인이다.

## 1) 서비스 방향과 목적

### 1.1 서비스 목적

- 모청은 사용자가 **쉽고 빠르게 모바일 청첩장**을 만들고 공유할 수 있게 하는 서비스다.
- 핵심 가치는 `제작 편의성`, `즉시 공유성`, `모바일 가독성`, `낮은 진입장벽`이다.

### 1.2 링크/유지 정책

- 청첩장 발행 시 URL은 기본 **3~6개월 유지**를 목표로 한다.
- 공유받은 사용자는 해당 URL을 통해 언제든지 청첩장을 확인할 수 있어야 한다.
- 링크 접근 안정성(만료 정책, 리다이렉트, 비공개 처리)은 운영 정책과 동기화한다.

### 1.3 서비스 구조(1차)

- 1차 서비스 화면은 단순 구조를 원칙으로 한다.
  - 메인
  - 에디터
  - 간편회원(간편로그인 중심)
  - 마이페이지
  - 문의
- 회원가입 폼 기반의 복잡한 가입 절차 대신, **간편로그인 기반 인증**을 우선한다.

### 1.4 확장 방향(2차+)

- 1차 핵심 도메인은 모바일 청첩장이다.
- 이후 확장 도메인으로 아래 이벤트 템플릿을 단계적으로 지원한다.
  - 돌잔치
  - 부고 소식
- 확장 시에도 기존 편집 경험(옵션 구조/미리보기 일치성)을 유지한다.

## 2) 제품 원칙

- **설정-미리보기 일치성**: 에디터에서 보이는 옵션 상태는 미리보기에 즉시 반영되어야 한다.
- **의미와 동작 일치**: 옵션 라벨/아이콘/체크 상태는 실제 동작과 반대로 동작하면 안 된다.
- **기본값 우선 정책**: 사용자가 아무 설정을 안 해도 자연스럽고 안전한 결과를 제공한다.
- **종속 옵션 명확화**: 부모 옵션 OFF 시 하위 옵션 노출/동작을 제한한다.
- **수정 비용 최소화**: 텍스트/레이아웃/기능 변경은 작은 단위로 분리하고, 영향 범위를 명확히 한다.

## 3) UI 정책

### 3.1 에디터 패널 정책

- 섹션 내부 필드 배치는 `FormItem` 단위로 관리한다.
- 같은 수준의 옵션은 같은 계층 `FormItem`으로 분리한다.
- 종속 옵션은 부모 옵션 아래에 두고, 부모 OFF 시 숨긴다.
- 구분선은 공통 여백을 유지한다. (현재 기준: `my-2`)

### 3.2 라벨/아이콘 정책

- 메뉴 라벨 변경 시 대응 아이콘도 함께 변경한다.
- 라벨 문구는 기능을 직접 설명해야 하며 추상 표현을 피한다.
- 용어는 서비스 전반에서 통일한다. 예: `다국어 지원`, `정보순서`, `노출 수`.

### 3.3 시각 톤 정책

- 미리보기의 배경 톤은 실제 서비스 톤보다 과하지 않게 약화해 가독성을 우선한다.
- 테마 포인트 컬러(`--key`)는 배경보다 상대적으로 명확히 보이도록 유지한다.
- 파티클/강조 배경 등 시각 효과는 본문 가독성을 훼손하지 않아야 한다.

## 4) UX 정책

### 4.1 즉시 반영 UX

- 옵션 변경 시 저장 버튼 없이 미리보기에 즉시 반영한다.
- 토글형 옵션은 ON/OFF 결과가 명확하게 구분되어야 한다.
- 하위 옵션은 부모 옵션 활성 상태에서만 조작 가능/노출 가능해야 한다.

### 4.2 피드백 UX

- 사용자 액션(복사, 저장 등)은 즉시 피드백을 제공한다.
- 복사 성공/실패는 미리보기 내부 토스트로 안내한다.
- 피드백 메시지는 구체적으로 작성한다. 예: `계좌정보가 복사되었습니다.`

### 4.3 정보 노출 순서 UX

- `신부측 정보 먼저 노출` 옵션은 이름/부모/연락처/관련 문구까지 일관되게 반영한다.
- 일부 블록만 순서가 바뀌는 불일치를 허용하지 않는다.

## 5) 서비스 기능 정책

### 5.1 옵션 종속성 정책

- 예식정보:
  - `달력 사용` OFF면 달력 UI는 숨긴다.
  - `D-Day 표시` ON이면 달력 OFF여도 D-Day 텍스트는 표시 가능하다.
  - `배경컬러 노출`은 `달력 사용` ON에서만 노출된다.
- 갤러리:
  - `레이아웃 = 슬라이드`일 때만 `이미지 자동 전환`, `전환시간`을 노출한다.
  - `레이아웃 = 그리드`일 때만 그리드 관련 옵션을 노출한다.

### 5.2 미디어/에셋 정책

- 사용자 시각 효과용 리소스는 `public/`에서 관리한다.
- 파일명은 ASCII 기반 표준명으로 정규화한다. (예: `petal01.svg`)
- 외부 리소스 교체 시 기존 참조 경로와 신규 자산 목록을 함께 검증한다.

### 5.3 파티클 정책

- 파티클은 성능과 가독성을 우선하며 과도한 개수를 피한다.
- 효과별 기본값(개수/크기/속도)은 목적에 맞게 차등 적용한다.
- 꽃잎 효과:
  - 지정 SVG 풀에서 랜덤 선택
  - 각도/속도 랜덤
  - 화면 밖에서 자연스럽게 유입
- 눈송이/반짝임/하트:
  - 테마 컬러 연동
  - 노출 강도(개수/크기) 일관 정책 적용

## 6) 백엔드/데이터 정책

### 6.1 데이터 모델 정책

- 신규 옵션 추가 시 저장 키를 명시하고 기본값을 정의한다.
- 기본값은 렌더 단계에서 안전하게 보정한다.
  - 예: `?? true`, `?? false`, 허용 값 집합 검증
- 미리보기 계산 값은 상태에서 파생되며, 상태 자체를 임의 변경하지 않는다.

### 6.2 입력/보안 정책

- URL/파일 입력은 클라이언트에서 최소 검증 후 저장한다.
- 비밀번호는 UI에서 평문 재노출하지 않는다.
- 민감 액션(수정/삭제)은 재확인(비밀번호 등) 절차를 유지한다.

### 6.3 복사/클립보드 정책

### 6.4 인증/계정 정책

- 인증은 간편로그인 중심으로 운영하며, 별도 회원가입 절차를 최소화한다.
- 인증된 사용자는 즉시 에디터 및 마이페이지 기능을 이용할 수 있어야 한다.
- 계정 식별/세션 만료/재인증 흐름은 사용자의 작성 중단을 최소화하도록 설계한다.

- Clipboard API 실패를 대비한 fallback을 제공한다.
- 실패 시에도 사용자에게 실패 토스트를 명확히 노출한다.

## 7) 개발 프로세스 정책

### 7.1 변경 단위

- 하나의 사용자 요청은 가능한 한 한 파일/한 섹션 단위로 최소 수정한다.
- 구조 변경 시 기존 기능 회귀(라벨/순서/종속성)를 함께 점검한다.

### 7.2 검증 기준

- 필수: 변경 파일 린트 확인
- 권장: `npm run build`로 타입/빌드 검증
- 옵션/토글 변경 시 미리보기 동작 수동 확인

### 7.3 커밋 기준

- 커밋 메시지는 변경 의도 중심으로 작성한다.
- UI 구조 변경 + 동작 변경이 함께 있을 경우 하나의 의미 단위로 묶는다.
- 푸시 전 워킹트리가 깨끗한지 확인한다.

## 8) 접근성 및 국제화 정책

- 토글/버튼은 키보드 입력(Enter/Space)을 지원한다.
- `aria-label` 등 보조 기술용 속성은 기능성 액션에 우선 적용한다.
- 다국어 관련 옵션은 독립적인 섹션으로 유지하고, 사용자 선택이 실제 노출 순서/문구에 반영되어야 한다.

- 회귀 방지를 위해 핵심 옵션 조합에 대한 E2E 테스트를 추가한다.
