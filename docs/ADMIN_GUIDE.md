# 모청 관리자(마스터) 운영 매뉴얼

본 문서는 모청 서비스의 **관리자(ADMIN) 계정 운영 기준**과 **일상 운영 절차**를 정의한다.
상위 기준 문서는 `docs/PRODUCT_STANDARD.md`, 백엔드 스펙은 `docs/BACKEND_MVP_SPEC.md`를 참조한다.

---

## 1. 기본 원칙

- 관리자 계정은 **별도의 가입 절차 없이** 일반 소셜 로그인으로 로그인한 계정을 **승격**해 만든다.
- 내부적으로 한 이용자 = 1개의 `User.id`를 유지한다. 관리자라는 이유로 별도 계정을 만들지 않는다.
- 관리자 계정도 동일하게 세션/감사로그 정책의 적용을 받는다.
- 관리자 권한은 **필요한 사람에게만 최소한으로** 부여한다.

---

## 2. 관리자 권한 범위

현재 구현 기준(`/admin` 및 관련 가드)에서 관리자가 할 수 있는 것:

- 전체 회원 수, 전체 초대장 수, 결제 완료 건수 확인
- 최근 결제 20건 조회 (사용자 이메일, 주문번호, 금액, 상태)
- 향후 확장(목표)
  - 사용자별 초대장/결제 상세 조회
  - 결제 강제 취소/환불 처리
  - 초대장 노출 중단/링크 비활성화
  - AuditLog 조회 UI

관리자 외 역할은 정의하지 않는다(단일 권한 모델).

---

## 3. 관리자 계정 승격 절차

### 3.1 최초 관리자 등록

1) 관리자가 될 이메일로 **서비스에 한 번 로그인**한다.
   - `/login` → Google/Kakao/Apple 중 하나로 로그인
   - 콜백 완료 시 `User` 테이블에 해당 계정이 자동 생성된다(이메일은 소문자 저장).
2) 서버(또는 로컬) 터미널에서 아래 명령 실행:

```bash
cd invitation-builder
npm run set-admin -- your@email.com
```

3) 성공 로그 확인:

```
[set-admin] OK: your@email.com 계정을 ADMIN 으로 설정했습니다.
```

4) 브라우저에서 `/admin` 접속 → 대시보드가 열리면 완료.

### 3.2 관리자 추가

- 동일 절차를 새 이메일로 반복 실행한다.

### 3.3 관리자 강등 (향후 작업)

- 현재는 스크립트에 USER 복귀 옵션이 포함되어 있지 않다.
- 임시로 강등이 필요한 경우 DB에서 직접 `User.role = 'USER'`로 업데이트한다.
- 공식 절차로 승격/강등 모두를 다룰 수 있게 `set-admin` 스크립트를 `--role` 옵션으로 확장하는 것이 후속 과제다.

### 3.4 자주 발생하는 실패 케이스

- `set-admin`이 “계정을 찾을 수 없습니다”로 실패
  - 아직 한 번도 로그인하지 않은 이메일이다. **먼저 `/login`에서 로그인**해 `User` 레코드가 생성되게 한다.
- 이메일 대소문자 불일치
  - 스크립트는 입력을 소문자로 정규화한다. DB 역시 소문자로 저장되므로 대소문자 불일치로 실패하지 않는다.
- 같은 이메일을 여러 Provider로 사용
  - 계정 병합 정책(`PRODUCT_STANDARD.md §3.2`)에 따라 같은 `User` 행으로 병합되므로 한 번의 승격이면 충분하다.

## 4. 관리자 화면 (`/admin`)

### 4.1 접근 가드

- 라우트: `invitation-builder/app/admin/page.tsx`
- 가드: `requireAdmin()` → 비로그인 시 `/login?next=/admin`, 일반 사용자는 `/mypage?forbidden=admin`로 리다이렉트.
- `middleware.ts`가 `/admin` 경로에 대해 세션 검사를 1차로 수행.

### 4.2 구성 요소(현재)

- 상단 3-카드 요약
  - 전체 회원
  - 전체 초대장
  - 결제 완료 건수
- 최근 결제 목록 (최대 20건)
  - 사용자 이메일
  - 주문번호, 생성일
  - 금액, 상태

### 4.3 확장 예정

- 필터 (상태별/기간별)
- 사용자 상세 모달 (소유 초대장/결제 이력)
- AuditLog 검색 화면

## 5. 운영 루틴

### 5.1 일일 점검 (권장)

- `/admin` 접속 → 결제 상태에 `FAILED` / `REFUNDED` 스파이크 여부 확인.
- DB 알림/오류 모니터링 툴에 결제/인증 이벤트 경보가 떠 있는지 확인.
- 로그인 실패율 급증 시 Provider 설정(Redirect URL, Client Secret 만료 등) 우선 점검.

### 5.2 주간 점검

- AuditLog에서 관리자 이벤트(`auth.login` / `auth.logout`) 확인.
- 관리자 계정 목록이 “현재 운영 중인 관리자 리스트”와 일치하는지 점검.
- 환경변수 만료/교체 이슈(Provider Secret, PG Key 등) 주기 확인.

### 5.3 분기 점검

- 관리자 권한 보유자 재평가(유지/강등).
- URL 만료 정책, 결제 트리거, 무료 플랜 한도 등 `PRODUCT_STANDARD.md §12` Open Questions 재검토.

## 6. 장애 대응 매뉴얼 (요약)

### 6.1 로그인 전체 실패

1) 증상: 모든 사용자가 로그인 시 콜백에서 실패 또는 `/login?error=...`로 복귀.
2) 점검 순서
   - `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` 값 유효성 확인.
   - Supabase Dashboard → Auth → Providers → 각 Provider 활성화 상태 및 Client Secret 만료 여부.
   - Redirect URL 등록 여부(`<APP_URL>/auth/callback`).
   - `NEXT_PUBLIC_APP_URL` 이 실제 서비스 URL과 일치하는지.
3) 복구
   - 값 교정 → 서버 재시작 → 대표 관리자 계정으로 로그인 확인.

### 6.2 `/admin` 접근 불가

1) 증상: 관리자가 `/admin` 접근 시 `/mypage?forbidden=admin`으로 이동.
2) 원인 후보
   - 해당 계정의 `role`이 `ADMIN`이 아님.
   - 세션이 다른 계정(예: 다른 브라우저 프로필) 쿠키를 사용 중.
3) 복구
   - `npm run set-admin -- <email>` 재실행.
   - 동일 브라우저에서 로그아웃 후 재로그인.

### 6.3 결제 상태 불일치

1) 증상: PG 결제는 성공으로 보이나 `Payment.status`가 `READY`에 머무름.
2) 원인 후보
   - Webhook 수신 실패(엔드포인트 접근/서명 검증 실패).
   - 환경 변경으로 콜백 URL 불일치.
3) 복구
   - Webhook 로그/재전송 기능으로 수동 재시도.
   - 상태 강제 갱신은 최후 수단이며 **반드시 AuditLog 남김**.

### 6.4 악용/이상 징후

- 동일 IP에서 다중 계정 반복 생성, 짧은 시간 내 결제 실패가 다수인 경우
  - 해당 사용자 계정 비활성화(향후 기능) 또는 DB에서 일시 제한.
  - AuditLog로 행위 시점 추적 후 조치 기록 남김.

---

## 7. 개인정보/보안 기준

- 관리자는 사용자의 개인정보를 운영 목적 외로 조회/복사하지 않는다.
- 관리자 화면에서 얻은 정보는 외부로 유출하지 않는다.
- 관리자 계정도 2차 인증(OTP) 도입 전까지는 **계정 Provider 측 2FA 사용을 권장**한다.
- 퇴사/이탈 인원의 관리자 권한은 **즉시 강등** 한다.
